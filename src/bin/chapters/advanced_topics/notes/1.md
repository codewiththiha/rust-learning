

# How RefCell Actually Works — Step by Step

## Your regular reference example **won't compile**:

```rust
let mut i = 12;
let n = &i;      // shared reference
*n = 5;           // ❌ ERROR: cannot assign through shared reference
```

Even `&mut` doesn't allow what you're describing:

```rust
let mut i = 12;
let n = &mut i;   // mutable borrow
*n = 5;            // ✅ works
let m = &mut i;   // ❌ ERROR: can't borrow again while `n` exists
```

## What RefCell Actually Does

```rust
use std::cell::RefCell;

let data = RefCell::new(12);
//  data is NOT declared mut!

// borrow_mut() returns a TEMPORARY smart pointer (RefMut)
// that auto-drops at end of statement

*data.borrow_mut() = 5;
// Step by step:
//   1. borrow_mut() → creates RefMut guard, marks "someone is mutating"
//   2. *guard = 5   → writes 5 into the data
//   3. guard DROPS   → marks "nobody is mutating anymore"

println!("{}", data.borrow()); // 5

*data.borrow_mut() = 7;
// Same thing:
//   1. borrow_mut() → creates NEW guard, marks "someone is mutating"
//   2. *guard = 7   → writes 7 into the data  
//   3. guard DROPS   → marks "nobody is mutating anymore"

println!("{}", data.borrow()); // 7
```

## The Guard System — This Is the Key

```rust
use std::cell::RefCell;

let data = RefCell::new(10);

// When you call borrow_mut(), you get back a GUARD object
{
    let mut guard = data.borrow_mut();  // guard CREATED → locked 🔒
    *guard = 20;
    *guard = 30;
    // you can use it multiple times
    println!("{}", *guard); // 30
}   // guard DROPPED here → unlocked 🔓

// Now someone else can borrow
{
    let mut guard2 = data.borrow_mut(); // new guard → locked 🔒
    *guard2 = 50;
}   // unlocked 🔓

println!("{}", data.borrow()); // 50
```

## Visualizing the Lock Mechanism

```
Timeline:
─────────────────────────────────────────────────────

                                    State: UNLOCKED 🔓

println!("{}", *r);                (reading: 30)
drop(r);                           State: UNLOCKED 🔓

*g2 = 50;

```

## What Happens If You Break the Rule

```rust
let data = RefCell::new(10);

let mut guard1 = data.borrow_mut();  // 🔒 locked

// guard1 is STILL ALIVE here...
let mut guard2 = data.borrow_mut();  // 💥 PANIC! already locked!

// This is the runtime equivalent of:
// let mut x = 10;
// let a = &mut x;
// let b = &mut x;  // ← compile error in normal Rust
```

## One-Liner vs Holding the Guard

```rust
let data = RefCell::new(10);

// ONE-LINER: guard is created and dropped in same statement
*data.borrow_mut() = 20;  // lock → write → unlock (instant)
*data.borrow_mut() = 30;  // lock → write → unlock (instant)
// ✅ These work because each guard dies immediately

// HOLDING THE GUARD: you control when it unlocks
let mut g = data.borrow_mut();  // lock
*g = 40;
*g = 50;
// guard still alive... nobody else can borrow
drop(g);  // NOW it unlocks
```

## With Rc — Multiple Owners, Same Rule

```rust
use std::rc::Rc;
use std::cell::RefCell;

let shared = Rc::new(RefCell::new(0));

let owner_a = Rc::clone(&shared);
let owner_b = Rc::clone(&shared);

// owner_a mutates (one-liner, guard drops immediately)
*owner_a.borrow_mut() = 100;   // lock → write → unlock

// owner_b mutates (safe because previous guard is gone)
*owner_b.borrow_mut() += 50;   // lock → write → unlock

// owner_a reads
println!("{}", owner_a.borrow()); // 150

// ALL THREE point to the SAME data:
//   shared ──┐
//   owner_a ─┼──→ RefCell { value: 150, borrow_state: unlocked }
//   owner_b ─┘
```
